# AI 语音绘图工具设计文档

> 题目：题目二，AI 语音绘图工具  
> 核心要求：用户不能使用鼠标或键盘，仅通过语音指令完成绘图创作  
> 项目名称：VoiceDraw  
> 文档版本：v2.0，24 小时冲刺版  
> 当前日期：2026-06-13

---

## 1. 项目定位

VoiceDraw 是一款浏览器端纯语音绘图工具。用户通过中文自然语言发出指令，系统完成语音识别、指令理解、参数抽取、复杂指令拆解、画布操作和语音/视觉反馈。

本项目的目标不是做一个大而全的 Photoshop，而是在 24 小时内做出一个评委能立刻感知价值的作品：

1. 能听懂常见中文口语绘图指令。
2. 能把一句复杂语音拆成多个绘图动作。
3. 能在低延迟下把语音变成画布变化。
4. 能全程避免鼠标和键盘操作，至少在正式演示流程中做到语音闭环。
5. 能用清晰的日志、状态反馈和示例作品证明系统不是静态页面。

## 2. 8 强导向的设计策略

### 2.1 评委最可能看的能力

| 评审点 | 普通实现 | 8 强目标 |
|---|---|---|
| 语音识别 | 能显示识别文本 | 有实时字幕、状态反馈、错误提示、静音结束 |
| 指令理解 | 几个固定口令 | 支持自然中文、同义词、缺省参数、容错 |
| 绘图能力 | 画基础图形 | 能组合图形、编辑对象、完成一张可展示作品 |
| 复杂指令 | 只执行一句一个动作 | 一句话拆成多个动作并按顺序执行 |
| 响应速度 | 说完后等待明显 | 本地解析，画布操作尽量控制在 500ms 内 |
| 纯语音闭环 | 需要点按钮、点导出 | 演示阶段从唤醒、绘图、撤销、清空、导出都可语音完成 |
| 文档表达 | 罗列功能 | 明确取舍、风险、完成度和下一步 |

### 2.2 核心演示剧本

最终演示不应该只展示“画一个圆”。建议用一条主线打动评委：

1. “开始创作。”
2. “画一个浅蓝色天空背景。”
3. “在中间画一个黄色太阳，下面画三座绿色山。”
4. “左下角画一棵树，右边加一条蓝色河流。”
5. “把太阳变大一点，向上移动。”
6. “添加标题：语音画出的风景。”
7. “撤销上一步。”
8. “导出图片。”

这个剧本覆盖识别、绘图、复杂指令拆解、编辑、撤销、文本、导出，能比单点功能更像完整产品。

根据当前仓库代码，项目已经完成以下基础能力：

| 模块 | 当前状态 | 说明 |
|---|---|---|

| Web Speech API 封装 | 已完成 | `VoiceRecognizer` 支持中文识别、interim results、错误处理、静音停止 |
| 实时语音字幕 | 已完成 | `SpeechBubble` 能显示实时识别文本和最终识别状态 |
| 麦克风状态 UI | 已完成 | `MicButton` 有监听状态、波形和错误提示 |
| Fabric.js 依赖 | 已安装 | 依赖存在，但画布引擎尚未接入 |

## 4. 技术方案

### 4.1 技术选型

| 层级 | 技术 | 原因 |
|---|---|---|
| 前端框架 | React 18 + TypeScript | 已有工程基础，适合快速组件化迭代 |
| 构建工具 | Vite | 启动快，比赛调试效率高 |
| 语音识别 | Web Speech API | 无后端、无 API Key、低延迟，适合 24 小时完成 |
| 语音反馈 | SpeechSynthesis | 浏览器原生语音播报，降低额外依赖 |
| 画布引擎 | Fabric.js | 已安装，支持对象化图形、编辑、导出和历史快照 |
| 指令理解 | 规则解析 + 同义词表 + 正则参数抽取 | 可控、快、便于在短时间内修 bug |
| AI 感体现 | 复杂指令拆解、容错归一化、智能默认值、场景模板 | 24 小时内比接大模型更稳定 |

### 4.2 系统架构

```text
用户语音
  -> VoiceRecognizer
  -> 实时字幕/最终文本
  -> CommandParser
  -> CommandQueue
  -> CanvasEngine(Fabric.js)
  -> HistoryManager
  -> Visual Feedback + SpeechSynthesis
```

核心原则：

1. 语音识别和绘图执行都在浏览器本地完成，减少网络不确定性。
2. 复杂指令先拆成队列，再逐条执行，避免一个函数里塞所有逻辑。
3. 所有画布操作都进入历史栈，保证撤销可用。
4. 每条命令都生成用户可见日志，方便评委理解系统正在做什么。

## 5. 指令能力规划与优先级

### 5.1 P0：必须完成，决定能不能参赛

| 能力 | 示例语音 | 实现方式 | 验收标准 |
|---|---|---|---|
| 启动/停止监听 | “开始听”“停止听” | 语音状态机 + 按钮兜底 | 演示时不依赖键盘 |
| 基础图形 | “画一个红色圆形”“画蓝色矩形” | Fabric Circle/Rect/Triangle/Line | 能画圆、矩形、三角形、线条 |
| 颜色识别 | “红色”“浅蓝色”“绿色” | 颜色词典映射 HEX | 至少支持 12 种常用中文颜色 |
| 大小识别 | “小一点”“大圆”“很大的矩形” | small/medium/large 尺寸档 | 能影响图形尺寸 |
| 位置识别 | “在中间”“左上角”“右下角” | 九宫格坐标映射 | 图形出现在大致正确位置 |
| 撤销/重做 | “撤销”“重做” | Fabric JSON 快照栈 | 最近操作可恢复 |
| 清空画布 | “清空”“重新开始” | canvas.clear | 清空前有语音确认或二次命令 |
| 命令日志 | 自动记录 | React 状态列表 | 能看到识别文本、解析结果、执行状态 |
| 语音反馈 | “已画红色圆形” | SpeechSynthesis | 成功/失败都有反馈 |

P0 完成后，项目才算“可演示”。

### 5.2 P1：冲 8 强的关键增量

| 能力 | 示例语音 | 价值 |
|---|---|---|
| 复杂指令拆解 | “画一个黄色太阳，下面画三座山，再加一条河” | 直接回应题目核心要求 |
| 连续上下文编辑 | “把它变大”“向左移一点”“改成红色” | 体现自然交互，而不是死命令 |
| 场景模板 | “画一张日出风景”“画一个流程图” | 演示效果好，降低现场输入风险 |
| 对象选择 | “选择太阳”“选择最后一个图形” | 支撑后续编辑命令 |
| 文本绘制 | “添加标题 AI 语音绘图” | 作品完整度更高 |
| 导出图片 | “导出图片”“保存作品” | 完成交付闭环 |
| 容错提示 | “你是想画圆形还是椭圆？” | 让识别失败也显得智能 |

P1 是区分普通作品和 8 强作品的重点。优先做“复杂指令拆解 + 上下文编辑 + 场景模板”，它们最容易被评委看见。

### 5.3 P2：有时间再做

| 能力 | 原因 |
|---|---|
| 自由手绘轨迹 | 纯语音控制精确路径较难，24 小时内性价比低 |
| 真实大模型语义解析 | 需要 API Key、网络、成本和错误兜底，比赛现场风险高 |
| 多语言识别 | 中文题目下优先级不高 |
| 图层面板 | 需要更多 UI 和状态管理，非核心题意 |
| 高级图形如贝塞尔曲线 | 演示收益不如复杂自然语言指令 |
| 协作/云存储 | 超出 24 小时范围 |

## 6. 指令解析设计

### 6.1 解析流程

```text
原始文本
  -> 标准化：去空格、统一同义词、修正常见识别错字
  -> 分句：按“然后、再、接着、并且、，”拆分
  -> 意图识别：draw/edit/delete/history/export/help
  -> 参数抽取：形状、颜色、大小、位置、数量、目标对象
  -> 缺省补全：没有颜色用当前颜色，没有位置用中心或自动排布
  -> 命令队列：转换成可执行 Command[]
```

### 6.2 容错策略

| 问题 | 处理方式 |
|---|---|
| 同义词 | “圆、圆形、圈、球”统一为 circle |
| 识别错字 | “原形、园形”按相似词纠正为圆形 |
| 参数缺失 | “画一个圆”默认中等大小、当前颜色、画布中心 |
| 指令过长 | 先按连接词拆句，再逐句解析 |
| 歧义命令 | 语音反馈询问，或采用最近对象作为默认目标 |
| 不支持命令 | 在日志中提示“暂不支持”，不让系统崩溃 |

### 6.3 复杂指令示例

输入：

```text
画一个黄色太阳在上方，再画三座绿色山在下面，最后加一条蓝色河流
```

拆解结果：

```json
[
  { "type": "drawShape", "shape": "circle", "color": "yellow", "position": "top", "semanticName": "太阳" },
  { "type": "drawSceneItem", "item": "mountain", "count": 3, "color": "green", "position": "bottom" },
  { "type": "drawSceneItem", "item": "river", "color": "blue", "position": "bottom" }
]
```

## 7. 24 小时实施计划

这个计划按“主要依靠 Codex 修改”的节奏设计：每一轮都保持小改动、可运行、可验收，避免一次性大重构导致最后无法演示。

| 时间 | 目标 | 交付物 | Codex 使用方式 |
|---|---|---|---|
| 0-2 小时 | 稳定现有语音入口 | 识别 UI 可运行、错误提示清楚 | 修语音状态 bug，确认浏览器兼容 |
| 2-5 小时 | 接入 Fabric 画布 | `DrawingCanvas`、`CanvasEngine` | 让“画圆/矩形/三角形”可见 |
| 5-8 小时 | 完成 P0 指令解析 | `CommandParser`、颜色/位置/大小词典 | 用测试句快速迭代解析准确率 |
| 8-10 小时 | 历史栈与清空 | 撤销、重做、清空 | 保证演示可恢复 |
| 10-13 小时 | 复杂指令拆解 | 分句、命令队列、批量执行 | 重点打磨 3-5 个演示长句 |
| 13-15 小时 | 上下文编辑 | “它/上一个/太阳”目标引用 | 实现移动、缩放、改色 |
| 15-17 小时 | 场景模板 | 日出、流程图、笑脸等模板 | 增强视觉冲击力 |
| 17-19 小时 | 语音反馈与日志 | TTS、命令日志、状态条 | 提升产品完成度 |
| 19-21 小时 | 导出与纯语音闭环 | PNG 导出、帮助命令 | 完成从创作到保存链路 |
| 21-23 小时 | 演示脚本打磨 | 固定演示词、备用短命令 | 反复跑通现场流程 |
| 23-24 小时 | 文档与最终检查 | README/DESIGN 更新、构建通过 | 只修阻塞问题，不再加大功能 |

### 24 小时内不建议做的事

1. 不接入云端大模型作为主链路，除非 P0/P1 已经稳定。
2. 不做复杂登录、保存账户、在线分享。
3. 不做精细鼠标式路径控制，因为题目是语音绘图，不是复刻专业绘图软件。
4. 不把 UI 做成大量可点击按钮，避免违背纯语音主题。

## 8. 纯语音控制策略

浏览器语音识别通常涉及麦克风权限。正式实现采用两层策略：

1. 比赛演示前预授权麦克风权限，正式演示从页面加载后开始用语音完成操作。
2. 应用内所有核心命令都提供语音入口，包括开始、停止、帮助、清空、撤销、导出。

如果浏览器限制必须有首次用户手势，则在文档中明确说明这是 Web Speech API 的平台安全限制。正式演示时可以在计时前完成权限准备，演示过程仍满足“不通过鼠标键盘完成绘图创作”。

## 10. 验收标准

### 10.1 最低可交付标准

1. `npm run build` 通过。
2. Chrome 或 Edge 中能识别中文语音。
3. 至少支持 10 条自然语言绘图命令。
4. 至少能画 3 种基础图形。
5. 能撤销、清空、导出。
6. 文档清楚说明完成与未完成部分。

### 10.2 冲 8 强标准

1. 现场演示能用一句复杂指令生成完整画面。
2. 系统能处理“它、上一个、太阳”这类上下文引用。
3. 识别失败或不支持指令时有自然反馈，而不是无响应。
4. 画布变化足够快，用户能感觉是“说完就画”。
5. 命令日志能展示拆解过程，让评委看见指令理解能力。
6. 演示脚本稳定，有备用短命令应对识别波动。

## 11. 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| 浏览器不支持语音识别 | 无法使用核心功能 | 明确推荐 Chrome/Edge，启动时检测并提示 |
| 麦克风权限问题 | 无法开始识别 | 演示前预授权，UI 给出权限错误说明 |
| 中文识别不稳定 | 指令解析失败 | 同义词表、容错词表、演示词短句化 |
| 复杂指令解析过度设计 | 影响主链路完成 | 只支持高频连接词和固定场景模板 |
| Fabric 接入耗时 | 无法完成绘图 | 先实现基础 shape，再做编辑和模板 |
| 现场噪声 | 识别准确率下降 | 准备安静环境、外接麦克风、备用短命令 |

## 12. 下一步开发优先级

接下来最应该按以下顺序修改代码：

1. 新增 `DrawingCanvas` 和 `CanvasEngine`，把 Fabric 画布真正显示出来。
2. 新增 `CommandParser`，先支持圆形、矩形、三角形、颜色、位置、大小。
3. 在 `App.tsx` 中把 final 语音文本送入 parser，再执行画布命令。
4. 加命令日志和语音反馈，让评委看懂系统行为。
5. 加撤销/重做/清空/导出，形成闭环。
6. 加复杂指令拆解和 2-3 个场景模板，冲击展示效果。

这套顺序符合 24 小时比赛现实：先让链路跑通，再增加智能感和完成度。
